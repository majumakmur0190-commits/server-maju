Hasil Audit Kode PHP Folder API

Daftar Perbaikan API

Halaman ini berisi temuan yang layak diperbaiki setelah membaca seluruh file PHP di folder api. Fokusnya adalah bug nyata, inkonsistensi format API, risiko keamanan, dan file yang sebaiknya dipisah dari endpoint publik.

Prioritas Ringkas

Apa yang Paling Perlu Dikerjakan Dulu

Prioritas Tinggi

Keamanan dan file utilitas

  • Amankan atau hapus api/pass.php.
  • Nonaktifkan error detail di endpoint publik seperti login.php dan gambar.php.
  • Pindahkan kredensial database dari source code ke konfigurasi terpisah.
Prioritas Menengah

Standarisasi API

  • Samakan format request dan response semua endpoint.
  • Samakan nama field ID, status, dan error message.
  • Samakan pagination antar endpoint list.
Prioritas Rendah

Rapikan file lama

  • Rapikan nama file seperti sycn.php dan print copy.php.
  • Gabungkan endpoint yang fungsinya tumpang tindih.
  • Beri pemisahan jelas antara endpoint JSON, HTML, dan PDF.
Temuan Detail

Per File dan Per Masalah

Tinggi

api/pass.php

File ini langsung mengganti password user admin menjadi nilai tetap 12345 saat diakses. Ini berbahaya jika file bisa dibuka dari browser atau dari jaringan lokal.

  • Hapus file dari folder publik, atau pindahkan ke script maintenance yang tidak bisa diakses web.
  • Jangan simpan password default di source code.
  • Kalau masih dibutuhkan, lindungi dengan autentikasi admin dan token khusus.
Tinggi

api/barcode.php

Action get-all-barang menggunakan query yang salah:

SELECT * FROM barang WHERE limit 1

Query itu tidak valid dan sangat mungkin gagal saat dipakai.

  • Perbaiki menjadi query yang benar, misalnya SELECT * FROM barang LIMIT 1 atau query list sesuai kebutuhan.
  • Tambahkan status yang konsisten seperti endpoint lain.
  • Samakan nama action atau ganti file ini dengan endpoint yang lebih modern.
Tinggi

api/histori.php

Beberapa query masih menyisipkan nilai langsung ke SQL tanpa prepared statement. Ini meningkatkan risiko SQL injection dan membuat kode sulit dipelihara.

  • Ubah semua query dinamis ke prepared statement.
  • Validasi semua input numerik seperti id_pelanggan, id_barang, dan id_histori.
  • Tambahkan response HTTP code yang jelas untuk kasus invalid input.
Tinggi

api/login.php

Login masih menerima password plaintext jika isi kolom password sama persis. Ini menandakan sistem masih menoleransi password yang belum di-hash.

  • Paksa seluruh user memakai password hash saja.
  • Migrasikan password lama ke hash.
  • Matikan display_errors pada endpoint login di production.
Tinggi

api/gambar.php

Endpoint ini mengunduh gambar dari URL eksternal memakai file_get_contents(). Secara fungsi ini berjalan, tetapi rawan timeout, file besar, atau URL tidak aman.

  • Batasi domain atau minimal validasi URL dengan ketat.
  • Batasi ukuran file gambar maksimum.
  • Tambahkan timeout dan pengecekan MIME type.
  • Matikan debug error display di endpoint publik.
Menengah

api/sycn.php

Nama file kemungkinan salah ketik. File tetap berfungsi, tetapi nama seperti ini membingungkan dan berisiko membuat route salah di masa depan.

  • Ganti nama file menjadi sync.php.
  • Jika frontend sudah memakai nama lama, buat masa transisi atau redirect internal.
  • Dokumentasikan skema sinkronisasi dan konflik data.
Menengah

api/print copy.php

Nama file mengandung spasi. Secara teknis bisa dipanggil, tetapi kurang rapi dan merepotkan untuk routing, scripting, dan tooling.

  • Ganti nama menjadi print-copy.php atau hapus jika memang file percobaan.
  • Pastikan hanya satu versi file print yang dianggap aktif.
Menengah

api/order.php

File ini mengeluarkan HTML, bukan JSON, dan mengakses database lain yaitu gudang. Dari sudut desain API, file ini sebaiknya dipisah dari folder endpoint JSON.

  • Pindahkan ke folder halaman/report jika memang dipakai sebagai tampilan HTML.
  • Jika dibutuhkan oleh frontend modern, ubah output menjadi JSON.
Menengah

api/transaksi.php dan api/transaksi_scanner.php

Dua file ini sangat mirip. Risiko duplikasi bug cukup tinggi karena perubahan harus dilakukan di dua tempat.

  • Gabungkan logika bisnis penjualan ke satu service/helper.
  • Biarkan perbedaan hanya pada mode GET jika memang perlu.
  • Tambahkan validasi stok dan transaksi dalam fungsi bersama.
Menengah

api/import.php

File ini memakai koneksi database sendiri lewat PDO, berbeda dengan endpoint lain yang memakai db.php dan mysqli. Ini membuat konfigurasi jadi ganda.

  • Satukan konfigurasi koneksi database di satu tempat.
  • Tambahkan validasi field wajib selain nama_barang.
  • Pertimbangkan mode upsert jika import dipakai berulang.
Menengah

api/db.php

Kredensial database masih ditulis langsung di source code dan password database kosong.

  • Pindahkan konfigurasi ke environment file atau config luar web root.
  • Gunakan user database khusus aplikasi, bukan root jika memungkinkan.
  • Tambahkan mode error logging yang aman.
Rendah

Standarisasi Response

Format response belum seragam. Contohnya:

  • Sebagian memakai {status:"success"}.
  • Sebagian memakai {success:true}.
  • Sebagian mengembalikan array langsung tanpa wrapper.
  • File PDF dan HTML berada satu folder dengan endpoint JSON.

Saran standar minimal:

{
  "status": "success",
  "message": "Keterangan singkat",
  "data": {}
}
Rendah

Standarisasi Request

Beberapa endpoint memakai JSON body, beberapa form biasa, beberapa query action. Ini menyulitkan integrasi frontend.

  • Tentukan satu pola utama: REST JSON.
  • Kurangi endpoint model ?aksi=... dan ?action=... jika ada pengganti yang lebih bersih.
  • Samakan nama field ID seperti id atau barang_id sesuai konteks, lalu konsisten.
Saran Struktur Ulang

Usulan Rapikan Folder API

Masalah Sekarang Usulan
JSON API, PDF, HTML, dan utilitas campur dalam satu folder. Pisahkan menjadi api/, reports/, scripts/, dan storage/.
Duplikasi logika penjualan dan cetak. Pindahkan query dan logika hitung ke helper/service bersama.
Konfigurasi database ganda. Gunakan satu file konfigurasi pusat.
File sensitif masih bisa diakses web. Pindahkan file maintenance ke luar document root.
Kesimpulan

Status Umum

API aplikasi ini secara umum sudah berjalan dan endpoint utamanya cukup lengkap, tetapi ada beberapa hal yang memang layak diperbaiki. Yang paling penting adalah keamanan file utilitas, query yang salah di barcode.php, query non-prepared di histori.php, serta standarisasi format API agar frontend lebih mudah dipelihara.
